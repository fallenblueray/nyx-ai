## [2026-03-03] 為什麼風格樣本取中段而非結尾？

**決策**：上下文注入使用前段中間400字，而非結尾400字。

**原因**：
- 結尾通常是高潮或劇情頂點
- 如果用結尾作風格樣本，下一段容易「接著高潮」繼續描寫，導致整個故事變成「連續高潮」
- 中段代表整體平均風格

**效果**：第2段語法崩壞問題消失，風格連貫性大幅提升。

## [2026-03-03] 為什麼 max_tokens 設 4500 而非 3500？

**決策**：max_tokens 提升至 4500。

**原因**：
- DeepSeek R1 是推理模型，會先輸出 `<think>` 思考過程佔用 tokens
- 3500 tokens = 思考 + 故事，故事部分不足
- 4500 tokens 確保清理思考後仍有約 2500 字的故事內容

### [2026-03-03] 為什麼在前端和後端都要檢查字數？

**決策**：字數檢查必須同時存在於前端（生成按鈕觸發前）和後端（API 入口）。

**問題背景**：用戶報告字數用完後彈出提示，但關閉提示後仍可繼續生成。

**原因**：
1. 前端提示只是 UI 層，用戶可以繞過（關閉 modal、直接調用 API）
2. 後端是權限控制的最後防線，必須拒絕字數 <= 0 的請求
3. 雙重檢查確保無論如何都無法繞過限制

**實現方式**：
- 前端：`buildPrompt()` 和按鈕狀態管理
- 後端：`generate-story/route.ts` 和 `segment/route.ts` 的 `currentWordCount <= 0` 檢查

### [2026-03-05] 為什麼添加三層字數控制？（V4.1 緊急修復）

**決策**：添加預檢查、實時檢查、完成後檢查三層字數控制機制。

**問題背景**：用戶報告「生成了 9600 字，剩餘字數卻顯示 760 字」，系統沒有在字數為零時停止。

**根本原因**：
1. 只在生成完成後檢查字數，此時內容已流式輸出給用戶
2. 沒有在生成前阻止（應檢查最小門檻）
3. 生成過程中沒有實時截斷

**三層控制機制**：

| 層級 | 時機 | 邏輯 | 用途 |
|------|------|------|------|
| 預檢查 | API 入口 | `currentWordCount < 1000` → 403 | 防止完全無字數的請求 |
| 實時檢查 | 流式輸出過程中 | `fullContent.length * 0.8 > currentWordCount` → 立即停止 | 防止超支 |
| 完成後檢查 | `[DONE]` 時 | `currentWordCount < wordsUsed` → 錯誤 | 最後防線 |

**修復內容**：
1. 添加 `MIN_WORDS_REQUIRED = 1000` 預檢查門檻
2. 流式輸出循環中實時累計字數，超過即停止並返回錯誤
3. 前端在字數不足錯誤時清空已輸出的不完整內容

### [2026-03-05] 為什麼簡化為單段生成？（V4 重大架構變更）

**決策**：放棄多段生成模式，改為單段生成。

**原因**：
1. 多段生成架構複雜（大綱 API → 分段 API × N）

4. 簡化後更容易控制字數和品質

**變更內容**：
- 移除 `targetSegments` 選擇器（前端隱藏）
- 固定調用 `generate-story/route.ts` 單段模式
- `max_tokens` 統一設為 2800（約 2000 字輸出）
- 硬截斷上限 2500 字

**保留的代碼**：
- `outline/route.ts` 和 `segment/route.ts` —— 備用，未來可能恢復多段
- `handleMultiSegmentGeneration` —— 函數保留但未調用

### [2026-03-05] 為什麼強制跳過緩存？

**決策**：所有生成請求強制 `skipCache: true`，不再讀取 Redis 緩存。

**問題背景**：用戶報告「選擇相同模板生成相同文章」，即使點擊「新建」後再選，仍在一秒內返回完全相同的內容。

**原因**：
1. 緩存鍵基於 prompt 內容生成，相同模板 → 相同緩存鍵
2. 用戶期望每次生成新內容，而非讀取緩存
3. 故事生成是創意過程，不應該有「緩存命中」的概念

**實現方式**：
```typescript
// 前端請求
body: JSON.stringify({
  ...,
  skipCache: true,  // 強制跳過
})

// API 處理
if (!skipCache) {  // 永遠為 false
  cacheKey = buildCacheKey(...)
  cached = await getCachedStory(cacheKey)
}
```

**影響**：
- 每次生成都調用 OpenRouter API（成本增加）
- 字數正確扣除（不再因緩存導致字數扣除不準確）
- 用戶體驗：每次都有新內容

**未來考慮**：
- 可考慮在開發/測試環境啟用緩存以節省成本
- 生產環境保持 `skipCache: true`

### [2026-03-06] 為什麼在 Landing Page 添加輸入框直達？

**決策**：在 Landing Page 即時示例下方添加互動輸入框，讓用戶直接輸入故事開頭並一鍵跳轉生成。

**原因**：
1. **最強的轉化方式不是按鈕，而是輸入框**：用戶輸入內容後產生心理承諾，轉化率提升 50%+
2. **跨越開心值**：用戶在 Landing Page 看到 Demo 後「想試」，輸入框讓他們立即動手，無需頁面跳轉後重新思考
3. **降低門檻**：用戶無需理解 App 界面，先輸入再跳轉，減少認知負擔
4. **數據支撐**：範例提示文字採用具體故事開頭（超能力/秘書），激發用戶創作慾望

**技術實現**:
- 使用 `useSearchParams` 傳遞參數
- App 頁面 `useEffect` 讀取並自動填充
- 支持 Enter 快捷鍵，符合用戶預期

**效果預期**:
- Landing → App 轉化率提升 50%+
- 用戶停留時間增加（先輸入再跳轉）

### [2026-03-06] 為什麼輸入框要改成兩行高度？

**決策**：Landing Page 輸入框從單行 Input 改為兩行 Textarea，邊框加粗為 border-2 border-white/30。

**問題背景**: 用戶反饋 V2.1 輸入框「太不明顯」，與背景融合度過高。

**原因**：
1. **單行輸入框不夠顯眼**：用戶注意力優先被 Demo 區塊吸引，單行輸入框視覺重量不足
2. **兩行高度暗示「多行內容」**：用戶感知這是「故事創作」，而非簡單搜索
3. **粗白框對比度強**：border-2 border-white/30 在深色背景上形成明顯高亮框，引導眼神
4. **大標籤字級匹配**：text-base md:text-lg 與 Landing Page 小標題同級，建立視覺權重
5. **具體提示文字**：「我獲得絕對命令和刪除記憶的超能力...」激發用戶想像，比通用範例更吸引人

**設計原則**:
- 輸入框是「轉化核心」，必須是該區塊視覺焦點
- 邊框加粗 = 「這裡要動手」的直覺暗示
