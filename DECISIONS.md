## [2026-03-03] 為什麼風格樣本取中段而非結尾？

**決策**：上下文注入使用前段中間400字，而非結尾400字。

**原因**：
- 結尾通常是高潮或劇情頂點
- 如果用結尾作風格樣本，下一段容易「接著高潮」繼續描寫，導致整個故事變成「連續高潮」
- 中段代表整體平均風格

**效果**：第2段語法崩壞問題消失，風格連貫性大幅提升。

## [2026-03-03] 為什麼 max_tokens 設 4500 而非 3500？

**決策**：max_tokens 提升至 4500。

**原因**：
- DeepSeek R1 是推理模型，會先輸出 `<think>` 思考過程佔用 tokens
- 3500 tokens = 思考 + 故事，故事部分不足
- 4500 tokens 確保清理思考後仍有約 2500 字的故事內容

## [2026-03-03] 為什麼硬截斷上限 2800 字而非 2500 字？

**決策**：截斷上限定為 2800，目標是 2500。

**原因**：
- 2500 字是「目標」，給模型一些餘地
- 在 2800 附近尋找自然句子結束點截斷
- 這樣既不超標，又不會在句子中間硬切

## [2026-03-03] 為什麼大綱用 Kimi 而故事用 DeepSeek？

**決策**：大綱 = Kimi K2.5，故事生成 = DeepSeek R1。

**原因**：
- Kimi K2.5 速度快，JSON 輸出穩定，適合規劃
- DeepSeek R1 是推理模型，深度思考後輸出的故事細節豐富
- 兩者分工可以兼顧速度和質量

## [2026-03-01] 為什麼用規則摘要而非 AI 摘要？

**決策**：「繼續創作」時，用 story-utils.ts 的規則方法生成500字摘要，而非呼叫 AI。

**原因**：
- AI 摘要需要額外的 API 呼叫，慢且貴
- 用戶的故事可能超過 5000 字，導致「劇情主線太長」錯誤
- 規則方法：取末尾500字 + 提取角色名，既快又穩

### [2026-03-03] 為什麼在前端和後端都要檢查字數？

**決策**：字數檢查必須同時存在於前端（生成按鈕觸發前）和後端（API 入口）。

**問題背景**：用戶報告字數用完後彈出提示，但關閉提示後仍可繼續生成。

**原因**：
1. 前端提示只是 UI 層，用戶可以繞過（關閉 modal、直接調用 API）
2. 後端是權限控制的最後防線，必須拒絕字數 <= 0 的請求
3. 雙重檢查確保無論如何都無法繞過限制

**實現方式**：
- 前端：`buildPrompt()` 和按鈕狀態管理
- 後端：`generate-story/route.ts` 和 `segment/route.ts` 的 `currentWordCount <= 0` 檢查

### [2026-03-04] 為什麼提取自定義 hooks？

**決策**：將 StoryOutput.tsx 中的生成邏輯提取到 `useStoryGeneration` hook。

**原因**：
1. StoryOutput.tsx 已經超過 800 行，難以維護
2. 生成邏輯與 UI 渲染混在一起，違反單一職責原則
3. 提取後可以獨立測試生成邏輯
4. 其他組件可以復用這些邏輯

**提取內容**：
- SSE 數據處理邏輯
- 狀態管理（currentSegment, wordInfo 等）
- API 錯誤處理
- 生成流程控制（prepareGeneration, finishGeneration）

**預期收益**：
- StoryOutput.tsx 減少約 40% 代碼
- 生成邏輯可獨立測試
- 未來可以支持更多生成模式（如批量生成）

### [2026-03-03] 為什麼刪除 system prompt 的字數限制？

**決策**：從 system_prompt.ts 刪除「精準控制在4200～4800字之間的」這段描述。

**原因**：
1. 用戶現在可以選擇 1/2/3 段（對應 2500/5000/7500 字）
2. system prompt 的固定字數限制會與動態選擇衝突
3. 字數控制應該在 user prompt 和後處理中動態管理

**替代方案**：
- user prompt 動態計算並明確指定每段 2300-2500 字
- 後處理硬截斷到 2800 字作為保險
